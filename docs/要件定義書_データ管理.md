## 1. アプリの起動とデータの初期状態

### 1.1 初回起動時

#### 1.1.1 利用者マスタの状態

**初期状態**:
- 利用者マスタは**空の状態**
- LocalStorageに`users`キーが存在しない、または空配列

**表示内容**:
```
┌─────────────────────────────────────────┐
│                                         │
│     算定基礎を取り込んでください         │
│                                         │
│     [算定一覧から取り込む] ボタン       │
│                                         │
└─────────────────────────────────────────┘
```

**操作可能なボタン**:
- ✅ 算定一覧から取り込む（強調表示）
- ❌ 作業内容を取り込む（無効化）
- ❌ 作業内容を保存（無効化）
- ✅ ⚙️設定

#### 1.1.2 予定データの状態

**初期状態**:
- 予定データなし
- LocalStorageに`schedule_YYYY-MM`キーが存在しない

**制約**:
- データ0の状態では予定入力不可
- 利用者マスタがない状態では予定グリッドを表示しない

---

### 1.2 データ永続化の基本方針

#### 1.2.1 保存先

**LocalStorage**を使用

**理由**:
- ブラウザ標準機能（外部依存なし）
- オフライン動作可能
- シンプルな実装

#### 1.2.2 保持期間

**1ヶ月分のみ保持**

**ルール**:
- 現在編集中の月のデータのみ保持
- 他の月のデータは保持しない
- Excel保存により、事実上何ヶ月でも保存可能

**例**:
```javascript
// LocalStorageの状態（12月編集中）
{
  "users": [...],              // 利用者マスタ
  "schedule_2025-12": {...},   // 12月の予定のみ
  "notes": [...]               // 備考
}

// 11月に切り替えた場合
{
  "users": [...],              // 利用者マスタ（変更なし）
  "schedule_2025-11": {...},   // 11月の予定
  "notes": [...]               // 備考
}
// → schedule_2025-12 は削除される
```

#### 1.2.3 容量超過時の動作

**自動削除**:
- LocalStorage容量超過時、古い月を自動削除
- 警告表示は不要

**実装イメージ**:
```javascript
try {
  localStorage.setItem(key, data);
} catch (e) {
  if (e.name === 'QuotaExceededError') {
    // 最も古い schedule_YYYY-MM を削除
    deleteOldestSchedule();
    
    // 再試行
    localStorage.setItem(key, data);
  }
}
```

**削除対象**:
- `schedule_YYYY-MM` キーのみ
- `users` と `notes` は削除しない

## 2. 利用者マスタ管理

### 2.1 利用者データの構造

```javascript
{
  id: "user001",              // 内部ID（自動採番）
  name: "青柳美秋",           // 氏名
  registrationDate: "2025-01-01",  // 登録日
  sortId: 1,                  // 並び順ID
  note: "",                   // 利用者備考
  isActive: true              // 有効フラグ
}
```

#### 2.1.1 各フィールドの説明

| フィールド | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| id | string | ✅ | 内部ID（user001, user002, ...） |
| name | string | ✅ | 氏名 |
| registrationDate | string | ✅ | 登録日（YYYY-MM-DD） |
| sortId | number | ✅ | 並び順ID（0, 1, 2, ...） |
| note | string | ❌ | 利用者備考 |
| isActive | boolean | ✅ | 有効フラグ（true: 利用中、false: 削除済み） |

#### 2.1.2 内部IDの採番ルール

**形式**: `user` + 3桁ゼロ埋め番号

**採番方法**:
1. 既存の最大番号を取得
2. +1して新しいIDを生成

**例**:
```javascript
// 既存: user001, user002, user005
// → 次のID: user006

function generateUserId() {
  const users = loadUsers();
  const maxId = Math.max(...users.map(u => {
    const num = parseInt(u.id.replace('user', ''));
    return num;
  }));
  return `user${String(maxId + 1).padStart(3, '0')}`;
}
```

### 2.2 利用者の追加・削除・編集

#### 2.2.1 追加

**方法**:
- CSV取り込み時に自動追加
- 手動追加（将来的に対応予定）

**処理**:
1. 内部IDを自動採番
2. 登録日を現在日付で設定
3. sortIdを最大値+1で設定
4. isActiveをtrueで設定

#### 2.2.2 削除

**論理削除**:
- isActiveをfalseに変更
- データは保持

**理由**:
- 過去の予定データとの整合性
- 誤削除からの復旧

**処理**:
```javascript
function deleteUser(userId) {
  const user = findUser(userId);
  user.isActive = false;
  saveUsers();
}
```

#### 2.2.3 編集

**編集可能項目**:
- ✅ 氏名
- ✅ 利用者備考
- ✅ 並び順（sortId）

**編集不可項目**:
- ❌ 内部ID
- ❌ 登録日

**氏名変更時の注意**:
- CSV取り込み時の紐付けが機能しなくなる
- 変更時に警告を表示

```
┌─────────────────────────────────────────┐
│ 氏名を変更すると、次回のCSV取り込み時に│
│ 新しい利用者として認識されます。        │
│                                         │
│ 変更しますか？                          │
│ [変更する] [キャンセル]                 │
└─────────────────────────────────────────┘
```

### 2.3 利用者の並び順

#### 2.3.1 基本方針

**デフォルト**: 文字コード順（あいうえお順）

**カスタマイズ**: 手動並び替え可能

#### 2.3.2 sortIdの管理

**初期値**:
- CSV取り込み時：文字コード順に0, 1, 2, ...を付与
- 手動追加時：最大値+1

**更新**:
- ドラッグ＆ドロップで並び替え時に更新
- 0から連番で振り直し

**例**:
```javascript
// 並び替え前
{ id: "user001", name: "青柳美秋", sortId: 0 }
{ id: "user002", name: "安藤敏子", sortId: 1 }
{ id: "user003", name: "天野花子", sortId: 2 }

// 安藤を一番上に移動
{ id: "user002", name: "安藤敏子", sortId: 0 }
{ id: "user001", name: "青柳美秋", sortId: 1 }
{ id: "user003", name: "天野花子", sortId: 2 }
```

#### 2.3.3 月間での一貫性

**ルール**:
- どの月でも同じ順序で表示
- sortIdは利用者マスタで管理（月ごとに変わらない）

#### 3.3.1 利用者マスタが0件の場合

**処理フロー**:
```

2. 重複を除外（同じ利用者の複数行をまとめる）

4. 内部IDを自動採番（user001, user002, ...）

6. 利用者マスタに登録
```

**利用者確認画面を表示**:

```
┌──────────────────────────────────────────┐

├──────────────────────────────────────────┤
│ [同姓同名以外を全て取り込む]             │
├────┬──────┬────┬─────────────┤
│選択│氏名    │状態│操作              │
├────┼──────┼────┼─────────────┤
```

**判定ルール**:
- **既存**：利用者マスタに同名あり
- **新規**：CSVにあり、利用者マスタになし
- **削除候補**：利用者マスタにあり、CSVになし

**各状態のボタン有効/無効**:

| 状態 | [+]追加 | [✎]編集 | [🗑]削除 |
|------|---------|---------|---------|
| 既存 | ❌ | ✅ | ❌ |
| 新規 | ✅ | ❌ | ❌ |
| 削除候補 | ❌ | ❌ | ✅ |

**操作の説明**:

**[+]追加**（新規のみ）:
- 利用者マスタに追加
- 内部IDを自動採番

**[✎]編集**（既存のみ）:
- 氏名の手動修正（同姓同名の区別用）
- 例：「青柳美秋」→「青柳美秋(2)」

**[🗑]削除**（削除候補のみ）:
- 論理削除（isActive = false）

**チェックボックス**:
- ☑：取り込み対象
- ☐：取り込み対象外

#### 3.3.3 同姓同名の扱い

**CAREKARTEでの区別**:
- 区別記号付きで出力される
- 例：「青柳美秋」「青柳美秋(2)」

**実運用**:
- ほとんど発生しない
- 発生した場合のみ手動で対応

**「同姓同名以外を全て取り込む」ボタン**:
```javascript
// 同姓同名の判定
const nameCount = {};
csvUsers.forEach(user => {
  nameCount[user.name] = (nameCount[user.name] || 0) + 1;
});

// 同姓同名（2回以上出現）を除外
const filtered = csvUsers.filter(user => nameCount[user.name] === 1);
```

**同姓同名を取り込む場合**:
- 個別に[+]ボタンをクリック
- または[✎]ボタンで氏名を修正

#### 3.3.4 削除候補の処理

**削除候補とは**:
- 利用者マスタにあり、CSVになし
- 例：先月まで利用、今月は利用なし

**処理の選択肢**:

1. **削除する（論理削除）**:
   - [🗑]ボタンをクリック
   - isActive = false
   - 予定グリッドに表示されなくなる

2. **削除しない**:
   - チェックを外す（☐）
   - 利用者マスタに残る
   - 予定グリッドに表示される
   - 今月の予定は空白

**判断基準**:
- オペレーターが実運用に基づいて判断
- 例：一時的に利用なし → 削除しない
- 例：退所済み → 削除する

#### 3.3.5 既存スケジュールがある場合

**確認ダイアログ表示**:
```
┌─────────────────────────┐
│ 既存の予定データは上書き│
│ されます。               │
│ 備考は保持されます。     │
```

**動作**:
- 予定データ：**上書き**
- 備考（利用者備考・セル備考）：**保持**

**実装イメージ**:
```javascript
async function importCSV(files) {
  // 1. CSVパース
  const userPatterns = await parseCSVFiles(files);
  
  // 2. 既存スケジュールチェック
  const hasExisting = hasExistingSchedule(currentYearMonth);
  
  if (hasExisting) {
    const confirmed = await showConfirmDialog(
      '既存の予定データは上書きされます。\n' +
      '備考は保持されます。\n\n' +
      '算定基礎を取り込みますか？'
    );
    
    if (!confirmed) return;
  }
  
  // 3. 週間パターン→月間展開
  const calendar = expandWeeklyToMonthly(userPatterns, currentYearMonth);
  
  // 4. 備考を保持したまま上書き
  const mergedCalendar = mergeWithExistingNotes(calendar);
  
  // 5. 保存
  saveSchedule(currentYearMonth, mergedCalendar);
  
  // 6. UI更新
  renderGrid();
  
  showToast('算定基礎を取り込みました', 'success');
}
```

### 3.4 週間パターン→月間展開

#### 3.4.1 アルゴリズム（概要）

```
入力: 
  - userPatterns: 週間パターン（月〜日）
  - yearMonth: 対象月（例：'2025-12'）

処理:
  1. 対象月の全日付を取得（1〜31日）
  
  2. 各日付の曜日を判定（月〜日）
  
  3. 週間パターンから対応する値を取得
     - 通い: dayPattern[曜日]
     - 訪問: visitPattern[曜日]
     - 宿泊: stayPattern[曜日]
  
  4. セル値を設定
     - 通い: "○"
     - 訪問: 回数（"1", "2", ...）
     - 宿泊: 仮で"入"（後で調整）
  
  5. 連続する宿泊を「入所〜退所」に調整
     - 連続する日 → 開始日「入」、中間日「○」、終了日「退」
     - 単独の日 → 「入」と「退」を同日に

出力:
  - calendar: ScheduleCalendar形式の月間予定
```

**詳細な実装はExcelServiceのexpandWeeklyToMonthly()メソッドに委譲**

#### 3.4.2 宿泊期間の調整ロジック（概要）

```
入力: 
  - userCalendar: 利用者の月間カレンダー（仮の"入"が設定済み）

処理:
  1. 通泊行のセルを日付順にソート
  
  2. 連続する宿泊日をグループ化
     - 日付の間隔が1日 → 連続
     - 日付の間隔が2日以上 → 別グループ
  
  3. 各グループに対して
     - 開始日 → "入"
     - 中間日 → "○"（通い+宿泊）
     - 終了日 → "退"
  
  4. 単独の宿泊日（前後に宿泊なし）
     - 開始日 → "入"
     - 終了日（同日） → "退"

出力:
  - 調整済みのuserCalendar
```

**詳細な実装はScheduleCalendarのcalculateStayPeriods()メソッドに委譲**

### 4.2 利用者マスタのJSON形式

```javascript
{
  id: "user001",
  name: "青柳美秋",
  registrationDate: "2025-01-01",
  sortId: 1,
  note: "毎月第3週は家族旅行",
  isActive: true
}
```

**保存キー**: `users`

**保存方法**:
```javascript
localStorage.setItem('users', JSON.stringify(users));
```

**読み込み方法**:
```javascript
const users = JSON.parse(localStorage.getItem('users') || '[]');
```

---

### 4.3 月間スケジュールのJSON形式

**保存キー**: `schedule_YYYY-MM`（例：`schedule_2025-12`）

**構造**:
```javascript
{
  "user001": {
    userId: "user001",
    yearMonth: "2025-12",
    cells: {
      "2025-12-01_dayStay": { /* ScheduleCell */ },
      "2025-12-01_visit": { /* ScheduleCell */ },
      // ...
    },
    stayPeriods: [
      { /* StayPeriod */ }
    ]
  },
  "user002": { /* ... */ }
}
```

**cellsのキー形式**:
- 通常セル: `YYYY-MM-DD_cellType`
- 左特別セル: `YYYY-MM-prevMonth_cellType`
- 右特別セル: `YYYY-MM-nextMonth_cellType`

---

### 4.4 備考のJSON形式

**保存キー**: `notes`

**構造**:
```javascript
[
  {
    id: "note001",
    targetType: "cell",
    targetId: "user001_2025-12-05_dayStay",
    content: "この日は家族行事のため不可",
    createdAt: "2025-12-01T10:00:00Z",
    updatedAt: "2025-12-01T10:00:00Z"
  },
  {
    id: "note002",
    targetType: "user",
    targetId: "user001",
    content: "毎月第3週は家族旅行",
    createdAt: "2025-01-01T10:00:00Z",
    updatedAt: "2025-01-01T10:00:00Z"
  }
]
```

**targetTypeの値**:
- `"cell"`: セル備考
- `"user"`: 利用者備考

**targetIdの形式**:
- セル備考: `userId_date_cellType`
- 利用者備考: `userId`

---

### 4.5 データの相互関係

```
利用者マスタ（users）
  ↓ userId で紐付け
月間スケジュール（schedule_YYYY-MM）
  ├─ cells（セルデータ）
  │   ↓ userId + date + cellType で紐付け
  └─ stayPeriods（宿泊期間）
      ↓ userId で紐付け

備考（notes）
  ├─ 利用者備考: targetId = userId
  └─ セル備考: targetId = userId_date_cellType
```

**整合性チェック**:
- 月間スケジュールのuserIdが利用者マスタに存在するか
- 備考のtargetIdが対応するデータに存在するか
- stayPeriodsのstartDateとendDateが妥当か

### 4.6 LocalStorageの容量見積もり

#### 4.6.1 データサイズ

**利用者マスタ（29名）**:
```
1人あたり: 約150バイト
29名: 約4.4KB
```

**月間スケジュール（29名 × 33セル × 2行）**:
```
1セルあたり: 約300バイト
29名 × 33セル × 2行 = 1,914セル
約574KB
```

**備考（100件想定）**:
```
1件あたり: 約200バイト
100件: 約20KB
```

**合計**: 約600KB（1ヶ月分）

#### 4.6.2 容量制限

**ブラウザ別の制限**:
- Chrome, Edge, Firefox: 5〜10MB
- Safari (macOS): 5MB
- Safari (iOS): 2.5MB

**余裕**:
- 1ヶ月分: 600KB
- Safari (iOS)でも4ヶ月分保持可能
- → 1ヶ月のみ保持で十分
