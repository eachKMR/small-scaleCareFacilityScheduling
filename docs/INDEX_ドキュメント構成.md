### ⚠️ 重要な注意事項（必読）

#### 注意1: 「効率化の誘惑」に注意

あなたは優秀なプログラマーなので、こう考えるかもしれません：

```
「通いセクションの実装を見た。
 きっと泊まりも訪問も同じような構造だろう。
 であれば、5行の共通HTMLを作って、
 各タブで表示する行をJavaScriptで制御すれば効率的だ。」
```

### ❌ これは絶対にやらないでください

**理由**:
- 各セクションは**独立して設計**されている
- 業務の性質が異なるため、分離が必要
- 共通化すると将来の拡張が困難になる
- 設計担当が意図的に独立させている

**正しいアプローチ**:
- 実装指示書の範囲だけを実装する
- 他のセクションのことは考えない
- 共通化は設計担当が判断する

---

#### 注意2: 実装指示書の範囲を守る

**✅ 実装すべきこと**:
- 実装指示書の「✅ やること」セクションに書かれていること**だけ**

**❌ 実装してはいけないこと**:
- 実装指示書の「❌ やらないこと」セクションに書かれていること
- 実装指示書に書かれていない「改善」や「拡張」
- 「ついでに」「せっかくだから」という追加機能

**具体例**:

```markdown
実装指示書に書かれていること:
✅ やること: 通いセクション（#kayoi-section）に日別サマリーを追加
❌ やらないこと: 泊まりセクション、訪問セクションには追加しない
```

```javascript
// ❌ 悪い例（実装指示書の範囲を超えている）
// 泊まりセクションにも日別サマリーを追加してしまう
const tomariSection = document.querySelector('#tomari-section');
tomariSection.innerHTML += dailySummaryHTML; // ← やってはいけない

// ✅ 良い例（実装指示書の範囲だけ）
const kayoiSection = document.querySelector('#kayoi-section');
kayoiSection.querySelector('.calendar-header')
  .insertAdjacentHTML('afterend', dailySummaryHTML); // ← これだけ
```

**理由**:
- 設計担当が意図的に範囲を限定している
- 勝手に拡張すると設計思想が崩れる
- Phase 2以降で別途実装する予定がある

---

#### 注意3: エラーは無視しない

```javascript
// ❌ 絶対にやってはいけない例
try {
  // エラーが出るコード
  const dailySummaryUI = new DailySummaryUI(summaryElement, scheduleManager);
} catch (error) {
  // ❌ 何もしない（エラーを握りつぶす）
}
```

**✅ 正しいアプローチ**:
```javascript
try {
  // エラーが出るコード
  const dailySummaryUI = new DailySummaryUI(summaryElement, scheduleManager);
  console.log('✅ 日別サマリー初期化成功');
} catch (error) {
  console.error('❌ 日別サマリー初期化エラー:', error);
  // エラーの詳細を記録
  // 設計担当に報告
}
```

**理由**:
- エラーを隠すと原因が分からなくなる
- デバッグが困難になる
- 他の機能への影響が見えなくなる

---

#### 注意4: 既存コードを変更しない

**❌ やってはいけないこと**:
```javascript
// 既存のイベントリスナーを削除
document.querySelector('#prev-month').removeEventListener('click', oldHandler);

// 既存のHTMLを削除
kayoiSection.innerHTML = ''; // ← 既存要素が消える

// 既存の関数を書き換え
function existingFunction() {
  // 既存の処理を削除して新しい処理を追加
}
```

**✅ 正しいアプローチ**:
```javascript
// 新しい機能を追加（既存は変更しない）
const summaryElement = document.querySelector('.kayoi-summary');
if (summaryElement) {
  const dailySummaryUI = new DailySummaryUI(summaryElement, scheduleManager);
}

// 既存機能はそのまま（必ず実行される）
document.querySelector('#prev-month')?.addEventListener('click', handlePrevMonth);
```

**理由**:
- 既存機能を壊すリスクがある
- 実装指示書は「追加」を指示している（「変更」ではない）

## 🎓 プロジェクトBの設計原則（共通）

設計担当・実装担当の両方が理解すべき原則です。

### 設計品質の5原則（前プロジェクトの教訓）

1. **曖昧さは実装時ではなく、設計時に排除する**
2. **データ型を明示する**（文字列かDateか？）
3. **インタラクションを詳細に記述する**（クリックしたらどうなる？）
4. **状態遷移を図示する**（どの状態からどの状態に遷移する？）
5. **テスト仕様書を先に書く**（何をテストすれば完成？）

詳細は `CHECKLIST_設計レビュー.md` を参照。

---

### 実装品質の原則（このプロジェクトで確立）

6. **LLM誤解予測レビューを実施する**（実装指示書作成前）
   - LLM実装者の「効率化の誘惑」を予測
   - 誤実装パターンを先回りして警告
   - 「やること」「やらないこと」「なぜか」を明記

詳細は `GUIDE_LLM誤解予測レビュー.md` を参照。

---

### 引継ぎの原則（このプロジェクトで確立）

7. **チャット切り替え時は引継ぎドキュメントを作成する**
   - 必ず読むべきドキュメント（読む順序）を明記
   - 読む理由を説明
   - 読まなくていいドキュメントも明示

詳細は `GUIDE_引継ぎドキュメント作成ルール.md` を参照。

---

### プロジェクトBの業務原則

1. **月別予定表の実務が唯一の基準** - CSV/Excelは入出力手段
2. **各セクションは独立** - 通い・泊まり・訪問は独立設計
3. **日付軸思考** - 「この日は誰？」に答える設計
4. **居室管理重視** - 泊まりセクションは居室が主役
5. **拡張性** - セクション追加・統合を前提
6. **CAREKARTEとの補完関係** - 予定調整と記録・請求の役割分担

## 📚 関連情報

### プロジェクトBとは

プロジェクトBは、小規模多機能型居宅介護の**利用調整業務**を支援するシステムです。

- **対象ユーザー**: 小規模多機能施設の調整担当者
- **主な機能**: 月別予定表、定員管理、空き状況の可視化
- **技術**: Webアプリケーション（ブラウザベース）
- **特徴**: 調整業務の属人性軽減

### 重要な注意事項

このプロジェクトは**完全に新規**のプロジェクトです。他のプロジェクトの情報は参照しません。
