## 1. プロジェクトBとCAREKARTEの役割分担

### 1.1 役割分担の表

| システム | 役割 | 目的 | 強み |
|---------|------|------|------|
| **プロジェクトB** | 予定調整 | 属人性の軽減、稼働率向上 | 俯瞰・可視化・支援 |
| **CAREKARTE** | 記録・請求 | 一気通貫、法令遵守 | 正確性・効率性 |

---

### 1.2 それぞれの役割

```
┌─────────────────────────┐
│ プロジェクトB            │
│ 【予定調整に特化】       │
│                          │
│ ✓ 月別予定表の俯瞰       │
│ ✓ 定員管理               │
│ ✓ 空き状況の可視化       │
│ ✓ 調整支援               │
└─────────────────────────┘
         ↓ データ連携
┌─────────────────────────┐
│ CAREKARTE                │
│ 【記録・請求に特化】     │
│                          │
│ ✓ 算定基礎               │
│ ✓ 利用票・提供票         │
│ ✓ 利用スケジュール       │
│ ✓ 請求データ生成         │
└─────────────────────────┘
```

---

### 1.3 補完関係

```
プロジェクトB（調整の視点）
  ＋
CAREKARTE（記録の視点）
  ＝
完全な業務支援
```

**重要**: 
- 互いの弱点を補う関係
- どちらか一方では不十分
- 両方あって初めて完結

---

### 1.4 CAREKARTEとは（簡潔に）

**介護記録・請求ソフト**の一つ。多くの小規模多機能施設で採用されている。

**主な機能**:
- 記録（介護記録、バイタル、連絡帳等）
- ケアプラン
  - **算定基礎**：利用者ごとの週間パターンを設定。利用票を最も手数少なく作るためのひな型
  - **利用票・提供票**：算定基礎から自動展開し、手修正で完成させる法令必須文書
- 請求（レセプト、国保連への送信）

**強み**:
- 記録から請求まで一気通貫
- 法令遵守（加算・減算の自動計算）
- 正確性・効率性

**弱み**:
- 月別予定表の俯瞰が難しい
- 空き状況がすぐわからない
- 定員管理が煩雑
- 調整支援機能がない

→ **プロジェクトBが補完**

## 2. データ連携の方法

### 2.1 データの流れ（片方向）

```
┌─────────────────┐
│ CAREKARTE       │
│ ・算定基礎      │
└─────────────────┘
       ↓ CSV出力（可能）
       ↓
┌─────────────────┐
│ プロジェクトB   │
│ ・月間予定調整  │
└─────────────────┘
       ↓ 画面表示
       ↓
  👁️ 人間が見ながら
       ↓ 手動転記
┌─────────────────┐
│ CAREKARTE       │
│ ・算定基礎修正  │
│ ・利用票作成    │
└─────────────────┘
```

**重要な制約**:
```
✅ CAREKARTE → プロジェクトB：CSV経由で可能
❌ プロジェクトB → CAREKARTE：CSVインポート不可
→ 手動転記のみ
```

---

### 2.2 CAREKARTEはCSVインポート機能を提供していない

**これは致命的な制約ではなく、設計思想**:

```
CAREKARTEの考え方：
「記録は人間が確認・入力すべき」
  ↓
自動取込ではなく、目視確認を重視
```

**プロジェクトBの対応**:
```
CSVエクスポート機能は不要
  ↓
代わりに「2画面並行作業」を前提とした設計
  ↓
画面表示を利用票形式に合わせる
```

---

### 2.3 算定基礎CSVのインポート

**CAREKARTE → プロジェクトB** の流れ：

```
【月末：翌月の予定調整開始】
1. CAREKARTEで「算定基礎CSV」を出力
   ↓
2. プロジェクトBにインポート
   ↓
3. 週間パターンを月間に自動展開
   ↓
4. 細かい調整を行う
```

**算定基礎CSVの構造**:
```csv
利用者名,時間,サービスコード,月,火,水,木,金,土,日
安藤,10:00-16:00,071111,1,-,1,-,1,-,-
安藤,朝,061111,-,1,-,1,-,-,-
```

**サービスコード**:
- `06xxxx` → 訪問（訪問セクション）
- `07xxxx` → 通い（通いセクション）
- `08xxxx` → 泊まり（泊まりセクション）

**週間パターン**:
- `1` = その曜日に利用
- `-` = その曜日は利用しない

**重要**:
- 算定基礎は「利用票を最も手数少なく作るためのひな型」
- 実際の月間予定は個別事情で変わるため、毎月手修正が必要
- **算定基礎のメンテナンスの難しさ**：実態が変化しても算定基礎が古いままだと、毎月大量の手修正が必要になる
- プロジェクトBは、この「週間パターン→月間展開→手修正」を支援し、実態とのズレを検出

---

### 2.4 2画面並行作業の前提

プロジェクトBは、**CAREKARTEと並べて使う**ことを前提に設計します。

```
┌────────────────┐  ┌────────────────┐
│プロジェクトB   │  │ CAREKARTE      │
│（参照画面）    │  │（入力画面）    │
└────────────────┘  └────────────────┘
      👁️               ⌨️
    見ながら         入力する
```

**設計方針**:
- プロジェクトBの画面は「参照しやすさ」を重視
- レイアウトは公的様式（利用票等）に準拠
- 印刷も想定

---

#### 印刷の設計方針

**理由**: いろんなユーザがいる

```
パターンA：2画面派
→ デュアルモニタで並べて作業

パターンB：紙派
→ プロジェクトBを印刷
→ 紙を見ながらCAREKARTE入力

パターンC：タブレット派
→ タブレットでプロジェクトB表示
→ PCでCAREKARTE入力
```

**設計方針**:
- A4横で印刷しやすいレイアウト
- 31日分を1枚または2枚に収める
- 白黒印刷でも見やすい

---

### 2.5 2つの作業場面

#### 場面1：算定基礎設定（月に1回程度）

**目的**: 利用実態に合った算定基礎を設定する

**なぜ重要か**:
```
当初：火・木通い（算定基礎に設定）
  ↓ 時が経過
実態：水木1泊2日、金曜通いが多数派に
  ↓ しかし
算定基礎：火・木通いのまま（古い）
  ↓ 結果
毎月大量の手修正が必要
```

**理想の算定基礎とは**:
- 「利用票を最も手数少なく作れるひな型」
- 実態が変化したら、算定基礎も修正すべき

**プロジェクトBの価値**（Phase 2）:
- 過去の実績を分析
- 実態と算定基礎のズレを検出
- **「気づき」を与える**：「水木1泊2日、金曜通いに変更すると手修正が減りますよ」
- 適切な算定基礎を推奨

```
CAREKARTEで算定基礎CSV出力
  ↓
プロジェクトBにインポート
  ↓
週間パターンを月間展開
  ↓
実績を分析（Phase 2）
  ↓
適切な算定基礎を推奨（Phase 2）
  ↓
CAREKARTEで算定基礎を修正
```

**Phase 1では**:
- CSV取込のみ実装
- 推奨機能は Phase 2

---

#### 場面2：利用票への転記（月初）

**目的**: プロジェクトBで調整した月間予定をCAREKARTEに反映する

```
プロジェクトBで予定確定
  ↓
利用票形式で画面表示
  ↓
CAREKARTEの利用票に手動転記
  ↓
利用スケジュール展開
  ↓
サービス提供開始
```

**重要**:
- プロジェクトBからのCSVエクスポートは不要
- 画面を見ながら手動転記
- 利用票形式の画面レイアウトが重要

---

### 2.6 連携のタイミング

| タイミング | 方向 | 内容 |
|-----------|------|------|
| **月末（25日頃）** | CAREKARTE → プロジェクトB | 算定基礎CSV出力・インポート |
| **月末（29日頃）** | プロジェクトB → CAREKARTE | 利用票への手動転記 |
| **月中（随時）** | 双方向 | 大きな変更時のみ両方更新 |

---

### 2.7 月中の変更対応

小規模多機能では、月の途中でサービス内容が頻繁に変わります。

#### パターンA：事前変更（数日前）

```
1. プロジェクトBで予定変更
   ↓
2. CAREKARTEの利用票を修正
   ↓
3. 利用スケジュールに反映
```

**例**: 「来週の水曜日、通いから泊まりに変更したい」

---

#### パターンB：当日変更

```
1. CAREKARTEで「実施/中止」を記録
   ↓
2. 余裕があればプロジェクトBも更新
   （次月の参考のため）
```

**例**: 「今日、急に体調不良で通いをキャンセル」

**基本方針**:
- 予定の調整 → **プロジェクトB**
- 実績の記録 → **CAREKARTE**
- 完璧に同期する必要はない

---

### 2.8 両システムでの更新ルール

```
【プロジェクトB】
役割：調整用のツール（下書き）
  ├─ 月が変わったらリセット可能
  ├─ 過去の実績は統計用に保持
  └─ あくまで「調整の道具」

【CAREKARTE】
役割：記録の正本
  ├─ 実績は必ず記録
  ├─ 請求の根拠になる
  └─ 永続的に保管
```

**重要**: 
- プロジェクトBは「下書き」
- CAREKARTEが「正式記録」
- この役割分担を明確にすることで、両システムの完璧な同期は不要

## 3. プロジェクトBでできること（CAREKARTEでできないこと）

**CAREKARTEの弱み**（記録・請求に特化しているため）:

```
× 月別予定表の俯瞰が難しい
× 定員管理が煩雑（手動カウント）
× 空き状況がすぐわからない
× 調整支援機能がない
```

**プロジェクトBの強み**（予定調整に特化）:

| 機能 | 内容 |
|------|------|
| **月別予定表** | カレンダー形式で月全体を俯瞰 |
| **定員管理** | 自動カウント、超過警告、前半後半区別 |
| **空き状況** | 日別の空き状況を即表示 |
| **調整支援** | 夜勤負担度（Phase 2）、履歴管理 |

→ **互いの弱点を補完**
