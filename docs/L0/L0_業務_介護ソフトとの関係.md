## 1. CAREKARTE（ケアカルテ）とは

### 1.1 CAREKARTEの概要

**CAREKARTE（ケアカルテ）**は、介護サービスの**記録と請求を一気通貫で行うシステム**です。

| 項目 | 内容 |
|------|------|
| **主な目的** | 介護保険の複雑な請求業務をシステム化 |
| **特徴** | 記録 → 請求 の一気通貫 |
| **提供** | 介護ソフトベンダー |

**ポイント**: 
```
日々のサービス提供の記録
  ↓
そのまま請求データにつながる
  ↓
これが理想（CAREKARTEが実現）
```

---

### 1.2 CAREKARTEの仕組み

CAREKARTEは、以下の**4つのステップ**で記録から請求までを一気通貫で処理します。

#### ステップ1：算定基礎

**算定基礎**とは、利用者ごとの**利用パターン（週間パターン）**を設定する機能です。

```
【役割】
月間予定のひな型を作る

【設定内容】
利用者：安藤さん
  月 火 水 木 金 土 日
通い：○ - ○ - ○ - -
訪問：- ○ - ○ - - -
泊まり：- - - - - ○ -

→ 「月・水・金は通い」「火・木は訪問」
```

**重要**: 
- これは「理想的な週間パターン」
- 実際は毎月手修正が必要

---

#### ステップ2：利用票・提供票

**利用票（第6表）・提供票（第7表）**は、**法令で求められる必須文書**です。

```
【作成方法】
算定基礎から自動展開
  ↓
手修正で完成
  ↓
利用者・各サービス事業所に配布
```

**利用票の構造**:
```
┌────────────────────────────────┐
│ サービス利用票（第6表）         │
│ 令和○年○月分                  │
├────────────────────────────────┤
│ 時間帯 │サービス│ 1  2  3 ... 31│
│        │        │月 火 水    日│
├────────────────────────────────┤
│10:00-  │通い    │ 1  -  1 ... -│
│16:00   │        │              │
├────────────────────────────────┤
│朝      │訪問    │ -  1  - ... -│
│        │        │              │
└────────────────────────────────┘

予定：月間の利用予定を記載
実績：（この段階では空欄）
```

---

#### ステップ3：利用スケジュール

**利用スケジュール**は、利用票に基づいて展開される**日々の記録**です。

```
【ステータスの流れ】
計画（利用票から自動生成）
  ↓
実施 / 中止（職員が毎日記録）
  ↓
請求データの元になる
```

**職員の作業**:
```
毎日の業務：
11/25（月）通い → ☑ 実施
11/26（火）訪問 → ☑ 実施
11/27（水）通い → ☐ 中止（体調不良）
```

**重要**: この実施/中止の記録が請求の根拠になる

---

#### ステップ4：請求データ

**請求データ**は、実施/中止の記録から**確立された手順で簡単に**生成されます。

```
【作成方法】
利用スケジュールの「実施」を集計
  ↓
請求データを生成
  ↓
国保連への請求
```

**注意**: 「ボタン一発」は誇張表現
- 実際には確認作業等が必要
- ただし、手作業に比べれば圧倒的に効率的

---

### 1.3 CAREKARTEの強み

| 強み | 内容 |
|------|------|
| **一気通貫** | 記録 → 請求が自動的につながる |
| **法令遵守** | 利用票・提供票等の必須文書を自動生成 |
| **請求の正確性** | 複雑な介護保険請求ルールに対応 |
| **効率化** | 手作業に比べて大幅に効率的 |

---

## 2. CAREKARTEでは対応できないこと

### 2.1 システムの限界

**重要な前提**:
```
システムは無限になんでもできるわけではない
  ↓
CAREKARTEは「記録と請求」に特化
  ↓
予定調整には不十分
```

---

### 2.2 予定調整業務の課題

CAREKARTEは記録と請求に最適化されているため、**予定調整の段階では使いにくい**。

#### 課題1：月別予定表の俯瞰が難しい

```
❌ できないこと：
月全体の空き状況が一目でわからない
誰がいつ泊まっているか俯瞰できない
```

#### 課題2：定員管理が煩雑

```
❌ できないこと：
同時利用者数の自動カウント
定員超過の警告
前半・後半の区別管理
```

#### 課題3：空き状況がすぐわからない

```
❌ できないこと：
「今日、泊まりに空きはありますか？」
→ 即答できない
```

#### 課題4：送迎ルート計算ができない

```
❌ できないこと：
送迎ルートのシミュレーション
移動時間の自動計算
タイムテーブルの可視化
```

#### 課題5：調整支援機能がない

```
❌ できないこと：
必要介助量の可視化
夜勤負担度の計算
リスクマネジメント支援
```

#### 課題6：営業支援機能がない

```
❌ できないこと：
空き状況チラシの自動生成
稼働率ダッシュボード
新規相談時のシミュレーション
```

---

### 2.3 なぜCAREKARTEでは難しいか

**理由**:
```
CAREKARTEの設計思想：
「記録をもとに請求データを生成する」
  ↓
予定調整は別の視点が必要：
「リソース配分」「空き状況の可視化」
「稼働率の最適化」「新規利用者獲得支援」
  ↓
異なる設計思想 = 異なるシステムが必要
```

**つまり**: CAREKARTEに予定調整機能を求めるのは無理がある

## 3. プロジェクトBとCAREKARTEの役割分担

### 3.1 それぞれの役割

```
┌─────────────────────────┐
│ プロジェクトB            │
│ 【予定調整に特化】       │
│                          │
│ ✓ 月別予定表の俯瞰       │
│ ✓ 定員管理               │
│ ✓ 空き状況の可視化       │
│ ✓ 送迎ルート計算         │
│ ✓ タイムテーブル         │
│ ✓ 調整支援・営業支援     │
└─────────────────────────┘
         ↓ データ連携
┌─────────────────────────┐
│ CAREKARTE                │
│ 【記録・請求に特化】     │
│                          │
│ ✓ 算定基礎               │
│ ✓ 利用票・提供票         │
│ ✓ 利用スケジュール       │
│ ✓ 請求データ生成         │
└─────────────────────────┘
```

---

### 3.2 役割分担の表

| システム | 役割 | 目的 | 強み |
|---------|------|------|------|
| **プロジェクトB** | 予定調整 | 属人性の軽減、稼働率向上 | 俯瞰・可視化・支援 |
| **CAREKARTE** | 記録・請求 | 一気通貫、法令遵守 | 正確性・効率性 |

---

### 3.3 2つのシステムの補完関係

```
プロジェクトB（調整の視点）
  ＋
CAREKARTE（記録の視点）
  ＝
完全な業務支援
```

**重要**: 
- 互いの弱点を補う関係
- どちらか一方では不十分
- 両方あって初めて完結

## 4. データ連携の方法

### 4.1 データの流れ（片方向）

```
┌─────────────────┐
│ CAREKARTE       │
│ ・算定基礎      │
└─────────────────┘
       ↓ CSV出力（可能）
       ↓
┌─────────────────┐
│ プロジェクトB   │
│ ・月間予定調整  │
└─────────────────┘
       ↓ 画面表示
       ↓
  👁️ 人間が見ながら
       ↓ 手動転記
┌─────────────────┐
│ CAREKARTE       │
│ ・算定基礎修正  │
│ ・利用票作成    │
└─────────────────┘
```

**重要な制約**:
```
✅ CAREKARTE → プロジェクトB：CSV経由で可能
❌ プロジェクトB → CAREKARTE：CSVインポート不可
→ 手動転記のみ
```

---

### 4.2 CAREKARTEはCSVインポート機能を提供していない

**これは致命的な制約ではなく、設計思想**:

```
CAREKARTEの考え方：
「記録は人間が確認・入力すべき」
  ↓
自動取込ではなく、目視確認を重視
```

**プロジェクトBの対応**:
```
CSVエクスポート機能は不要
  ↓
代わりに「2画面並行作業」を前提とした設計
  ↓
画面表示を利用票形式に合わせる
```

---

### 4.3 算定基礎CSVのインポート

**CAREKARTE → プロジェクトB** の流れ：

```
【月末：翌月の予定調整開始】
1. CAREKARTEで「算定基礎CSV」を出力
   ↓
2. プロジェクトBにインポート
   ↓
3. 週間パターンを月間に自動展開
   ↓
4. 細かい調整を行う
```

**算定基礎CSVの構造**:
```csv
利用者名,時間,サービスコード,月,火,水,木,金,土,日
安藤,10:00-16:00,071111,1,-,1,-,1,-,-
安藤,朝,061111,-,1,-,1,-,-,-
```

**サービスコード**:
- `06xxxx` → 訪問（訪問セクション）
- `07xxxx` → 通い（通いセクション）
- `08xxxx` → 泊まり（泊まりセクション）

**週間パターン**:
- `1` = その曜日に利用
- `-` = その曜日は利用しない

---

### 4.4 2画面並行作業の前提

プロジェクトBは、**CAREKARTEと並べて使う**ことを前提に設計します。

```
┌────────────────┐  ┌────────────────┐
│プロジェクトB   │  │ CAREKARTE      │
│（参照画面）    │  │（入力画面）    │
└────────────────┘  └────────────────┘
      👁️               ⌨️
    見ながら         入力する
```

**設計方針**:
- プロジェクトBの画面は「参照しやすさ」を重視
- レイアウトは公的様式（利用票等）に準拠
- 印刷も想定（いろんなユーザがいる）

## 6. 実務での運用フロー

### 6.1 月末～月初の流れ

```
【月末（25日頃）：翌月の予定調整】

1. CAREKARTEで算定基礎CSV出力
   ↓
2. プロジェクトBにインポート
   ↓
3. 週間パターン → 月間予定に自動展開
   ↓
4. プロジェクトBで細かい調整
   - 送迎ルート計算
   - タイムテーブル確認
   - 空き状況確認
   - 新規利用者の受入検討
   ↓
5. 予定確定（29日頃）
   ↓
6. プロジェクトBの「利用票形式表示」を開く


【月初（1～3日）：CAREKARTEへ反映】

7. プロジェクトBの画面を見ながら
   CAREKARTEの利用票に手動転記
   ↓
8. 利用票・提供票を確定
   ↓
9. 利用スケジュール展開（CAREKARTE）
   ↓
10. サービス提供開始


【月中：実績記録】

11. 職員が毎日「実施/中止」を記録（CAREKARTE）
    ↓
12. 大きな変更があれば
    - プロジェクトBも更新（次月の参考）
    - CAREKARTEの利用票も修正


【翌月初：請求処理】

13. CAREKARTEで請求データ生成
    ↓
14. 国保連へ提出
```

---

### 6.2 月中の変更対応

#### パターンA：事前変更（数日前）

```
1. プロジェクトBで予定変更
   ↓
2. CAREKARTEの利用票を修正
   ↓
3. 利用スケジュールに反映
```

#### パターンB：当日変更

```
1. CAREKARTEで「実施/中止」を記録
   ↓
2. 余裕があればプロジェクトBも更新
   （次月の参考のため）
```

**基本方針**:
- 予定の調整 → **プロジェクトB**
- 実績の記録 → **CAREKARTE**
- 完璧に同期する必要はない

---

### 6.3 両システムでの更新ルール

```
【プロジェクトB】
役割：調整用のツール
  ├─ 月が変わったらリセット可能
  ├─ 過去の実績は統計用に保持
  └─ あくまで「調整の道具」

【CAREKARTE】
役割：記録の正本
  ├─ 実績は必ず記録
  ├─ 請求の根拠になる
  └─ 永続的に保管
```

**重要**: 
- プロジェクトBは「下書き」
- CAREKARTEが「正式記録」
- この役割分担を明確に

---

## 7. データ連携の詳細

### 7.1 連携するデータ

#### 利用者マスタ

**CAREKARTE → プロジェクトB**:
```
基本情報：
  ├─ 氏名
  ├─ 被保険者番号
  ├─ 要介護度
  ├─ 生年月日
  └─ 住所（送迎用）
```

**プロジェクトB独自項目**:
```
調整用の情報：
  ├─ 必要介助量（泊まり用）
  ├─ 送迎要否
  ├─ 移動時間設定
  ├─ 訪問準備時間
  └─ その他調整用メモ
```

---

#### 算定基礎（週間パターン）

**CAREKARTE → プロジェクトB**:
```csv
利用者名,時間,サービスコード,月,火,水,木,金,土,日
安藤,10:00-16:00,071111,1,-,1,-,1,-,-
安藤,朝,061111,-,1,-,1,-,-,-
安藤,,081111,-,-,-,-,-,1,-
```

**プロジェクトBでの処理**:
1. CSVを読み込み
2. サービスコードで振り分け
   - `06xxxx` → 訪問予定
   - `07xxxx` → 通い予定
   - `08xxxx` → 泊まり予定
3. 週間パターンを月間に展開
4. 手動で調整

---

#### 月間予定

**プロジェクトB → CAREKARTE**:
```
利用票形式で画面表示
  ↓
人間が目視で確認
  ↓
CAREKARTEに手動入力
```

**CSVエクスポートは不要**:
- CAREKARTEが取り込めない
- 画面表示で十分

---

### 7.2 連携のタイミング

| タイミング | 方向 | 内容 |
|-----------|------|------|
| **月末（25日頃）** | CAREKARTE → プロジェクトB | 算定基礎CSV出力・インポート |
| **月末（29日頃）** | プロジェクトB → CAREKARTE | 利用票への手動転記 |
| **月中（随時）** | 双方向 | 大きな変更時のみ両方更新 |

---

## 8. プロジェクトBの存在意義

### 8.1 CAREKARTEに「足りない機能」

プロジェクトBは、CAREKARTEでは対応できない以下の機能を提供します：

| 機能 | CAREKARTEでは | プロジェクトBでは |
|------|--------------|-----------------|
| **月別予定表の俯瞰** | ❌ 難しい | ✅ カレンダー表示 |
| **定員管理** | ❌ 手計算 | ✅ 自動カウント・警告 |
| **空き状況** | ❌ わからない | ✅ リアルタイム表示 |
| **送迎ルート** | ❌ なし | ✅ 計算・シミュレーション |
| **タイムテーブル** | ❌ なし | ✅ 1日の詳細表示 |
| **必要介助量** | ❌ なし | ✅ 可視化・分析 |
| **営業支援** | ❌ なし | ✅ チラシ生成・ダッシュボード |
| **算定基礎推奨** | ❌ なし | ✅ 実績分析・推奨設定 |

---

### 8.2 ケアマネの声

**CAREKARTEだけだと**:
> 「予定調整の段階で全体が見えないと、後で記録を入力するときに困る」

> 「空きがあるのに断ってしまうのはもったいない」

> 「ベテランが異動したら、調整が回らなくなった」

**プロジェクトBがあれば**:
> 「月全体が一目でわかるから、調整が楽になる」

> 「空き状況がすぐわかるから、新規相談にも即答できる」

> 「システムが支援してくれるから、新人でも調整できる」

---

### 8.3 2つのシステムの価値

```
CAREKARTE：記録の専門家
  ├─ 日々の記録を正確に
  ├─ 請求を確実に
  └─ 法令を遵守

プロジェクトB：調整の専門家
  ├─ 月全体を俯瞰
  ├─ リソースを最適化
  └─ 属人性を軽減

→ 適材適所で使い分け
→ 両方あって初めて完結
```

### 10.1 CAREKARTEとは

記録 → 請求 を一気通貫で処理するシステム

強み：
  ✓ 正確性
  ✓ 効率性
  ✓ 法令遵守

限界：
  × 予定調整には不向き
```

予定調整に特化したシステム

強み：

  ✓ 調整支援
  ✓ 営業支援
  ✓ 属人性の軽減

  CAREKARTEとの2画面並行作業
```

片方向の流れ：

  ↓

  ↓ 手動転記

```

→ 利用票形式で表示、手動転記
```

プロジェクトB：
