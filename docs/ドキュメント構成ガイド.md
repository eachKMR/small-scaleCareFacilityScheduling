## 🔍 ドキュメントの探し方

### ケース1: 実装を始めたい

```
1. README.md を読む（プロジェクト概要）
2. 実装ガイド_INDEX.md を開く（現在のフェーズ確認）
3. 該当する実装ガイドを開く（実装開始）
```

---

### ケース2: 実装中に分からないことがある

```
実装ガイドに書いてある？
 ├─ YES → そのまま実装
 └─ NO → 未確定項目リストを確認
     ├─ 推奨案がある → 推奨案で実装
     └─ 推奨案がない → 設計担当に質問
```

---

### ケース3: クラスの詳細を知りたい

```
1. データモデル設計書_v3.0.md を開く
2. 該当するクラスのセクションを読む
3. より詳しい背景 → 要件定義書を参照
```

---

### ケース4: 仕様の背景を知りたい

```
1. 要件定義書_概要_v1.0.md を開く（全体像）
2. 該当する要件定義書を探す（詳細）
3. さらに詳しく → 設計書を参照
```

## 📝 ドキュメント更新ルール

### 誰が更新するか

| ドキュメント種別 | 更新者 |
|----------------|--------|
| 要件定義書 | 設計担当（重大な変更のみ） |
| 設計書 | 設計担当 |

### いつ更新するか

| タイミング | 更新対象 |
|----------|---------|

| 仕様変更時 | 要件定義書（必要に応じて）、設計書（必要に応じて） |

## 💡 ドキュメント活用のヒント

### 実装担当者向け

1. **実装ガイドを信頼する**
   - 実装ガイドに書いてあることは全て正しい
   - 迷った時は実装ガイドに戻る

2. **未確定項目リストを活用する**
   - 実装中の判断に困ったら参照
   - 推奨案を優先的に採用

3. **要件定義書は背景理解のため**
   - 実装の詳細は実装ガイドにある
   - 背景や理由を知りたい時に読む

---

### 設計担当者向け

1. **実装ガイドは具体的に**
   - コード例を含める
   - テスト項目を明示する

2. **未確定項目は早めに確定**
   - 実装中に困らないよう先回り
   - 推奨案を必ず提示

3. **完了報告をレビュー**
   - 次のフェーズの参考になる
   - 問題点を早期発見

---

### プロジェクトオーナー向け

1. **進捗はINDEXで確認**
   - 実装ガイド_INDEX.md が全体像

2. **意思決定は未確定項目リスト**
   - 決定すべき項目が整理されている
   - 推奨案を参考に判断

3. **成果は完了報告**
   - 各フェーズの成果が記録されている
